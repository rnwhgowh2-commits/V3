PART 1

프로그램 개발 프로세스

모든 소프트웨어가 거치는 5단계 사이클 — 기획에서 운영까지의 큰 흐름

왜 단계로 나누어 진행하나?
단계가 명확해야 지금 무엇을 결정해야 하는지가 분명해집니다. 코드부터 짜면 절반쯤 가서 "이거 누구를 위한 거였지?"가 되고, 운영을 무시하면 출시 직후에 무너집니다. 5단계는 순서이자 체크리스트입니다.
비유: 집을 지을 때 — 설계도 없이 벽돌부터 쌓지 않고, 입주 후 관리 계획 없이 집을 팔지 않습니다. 소프트웨어도 똑같이 단계가 있습니다.
단계핵심 질문결과물
1. 기획무엇을, 누구를 위해?요구사항 문서
2. 설계어떻게 구조를 잡을까?아키텍처·DB 설계도
3. 개발어떤 도구로 코드를 짤까?실행되는 코드
4. 배포사용자에게 어떻게 전달할까?운영 중인 서비스
5. 운영어떻게 더 좋게 만들까?개선된 서비스

각 단계에서 무엇을 결정하고, 무엇이 잘못되면 어떤 문제가 생기는지 칩을 눌러 확인하세요.

1. 기획 2. 설계 3. 개발 4. 배포 5. 운영
1
기획 — 무엇을, 누구를 위해
시작점
핵심 질문
누가 어떤 문제를 가지고 있고, 어떤 플랫폼(웹/앱)으로 풀까?
결정 사항
대상 사용자, 핵심 기능, 플랫폼, 일정·예산. 이 결정이 모든 기술 스택을 좌우
결과물
요구사항 문서, 화면 흐름도, 우선순위 목록
잘못되면
아무도 안 쓰는 기능을 열심히 만들게 됨
자동화 예시
"오픈마켓 가격을 1시간마다 자동 동기화한다" → 대상·주기·실패 처리 정의
2
설계 — 어떻게 구조를 잡을까
청사진
핵심 질문
화면(프론트)·로직(백)·데이터(DB)를 어떻게 나누고 연결할까?
결정 사항
시스템 아키텍처, DB 테이블 구조, API 명세, 사용할 기술 스택
결과물
아키텍처 다이어그램, DB ERD, API 문서
잘못되면
개발 중반에 구조를 갈아엎게 되어 처음부터 다시
자동화 예시
수집 스크립트 + DB 테이블 + 알림 모듈로 분리. Python+pandas+SQLite로 결정

개발 — 코드로 만든다
제작
핵심 질문
어떤 언어·프레임워크·라이브러리로 어떻게 짤까?
결정 사항
코드 작성, 단위 테스트, 코드 리뷰, 버그 수정
결과물
실행 가능한 코드, 테스트, Git 저장소
잘못되면
버그 폭탄, 유지보수 불가능한 코드

requests로 오픈마켓 API 호출 → pandas로 가공 → DB 저장 코드 작성

배포 — 사용자에게 전달
출시
핵심 질문
내 컴퓨터의 코드를 어떻게 사용자가 접속할 수 있는 곳에 올릴까?
결정 사항
서버 환경, 도메인, 보안, 모니터링 설정. Docker로 포장 → AWS에 배포
결과물
실제 운영 중인 서비스 URL, 배포 자동화 파이프라인
잘못되면
"내 컴퓨터에선 됐는데" 문제, 트래픽 폭증 시 다운
자동화 예시
스크립트를 AWS Lambda에 올려 cron으로 1시간마다 자동 실행
5
운영·개선 — 더 좋게 만든다
반복
핵심 질문
실제로 잘 작동하는가? 무엇을 더 개선할 수 있는가?
활동
버그 수정, 사용자 피드백 반영, 성능 모니터링, 신규 기능 추가, 보안 패치
결과물
개선된 서비스 → 다시 1단계로 돌아가는 사이클 형성
잘못되면
방치된 서비스는 빠르게 노후화 → 결국 폐기
자동화 예시
실행 로그 모니터링 → 오픈마켓 정책 변경 시 즉시 코드 수정 → 재배포
🔄 핵심 — 일직선이 아닌 사이클
5단계는 한 번 끝나고 마는 게 아니라 계속 반복됩니다. 운영하면서 발견한 새 요구사항이 다음 사이클의 기획이 됩니다. 이게 애자일의 본질입니다.
🎯 자동화 담당자 관점
비전공자가 처음 자동화를 만들 때 3번(개발)에만 매달리기 쉽습니다. 하지만 실무에서 진짜 어려운 건 1번(요구 정의)과 5번(운영)입니다. 가격·재고가 틀리면 안 되는 일이라 운영이 절반 이상의 노력을 차지합니다.

실무 기술 선택

실제 서비스가 만들어지는 흐름 — 기획부터 데이터베이스까지, "왜 이 기술을 쓰는가"

왜 플랫폼을 먼저 결정해야 할까?
플랫폼이 정해져야 어떤 언어·프레임워크를 쓸 수 있는지가 결정됩니다. 웹이냐 앱이냐에 따라 아예 다른 생태계가 열리고, 나중에 바꾸면 처음부터 다시 만들어야 합니다.
비유: 음식점을 열기 전에 "한식집이냐 카페냐"를 먼저 정해야 주방 설비·인테리어·직원 채용이 결정되는 것과 같습니다.

왜 프론트엔드에 3가지 언어가 필요한가?
화면 하나를 만들려면 구조(HTML) + 디자인(CSS) + 동작(JS) 세 가지가 모두 필요합니다. 셋이 각각 다른 역할을 담당하고, 브라우저가 이를 조합해 화면을 만듭니다.
비유: 집 짓기 — HTML은 설계도, CSS는 인테리어, JS는 전기·기계장치. 셋 다 없으면 집이 안 됩니다.
언어분류역할
HTML마크업구조 선언 (계산 불가)
CSS스타일꾸미기 (계산 불가)
JavaScript프로그래밍조건·반복·계산 모두 가능
💎 JS의 특이점
JavaScript는 Node.js를 통해 백엔드도 가능합니다. 유일하게 프론트+백 양쪽을 모두 커버하는 언어입니다.

왜 백엔드는 언어를 섞어서 쓰나요?
각 언어는 잘하는 분야가 다르기 때문입니다. 요리사도 파스타·스시·바비큐 장인이 따로 있듯, 기능마다 최적의 언어가 다릅니다. 서로 API로 소통하면 언어가 달라도 함께 동작합니다.
비유: 대형 병원에 내과·외과·응급실이 따로 있는 것처럼, 대형 서비스도 기능별로 전문 서버를 따로 둡니다. 이게 MSA(마이크로서비스)입니다.

실제 동작 원리

코드가 화면이 되기까지 — 웹·앱·서버·DB가 실제로 어떻게 연결되어 동작하는가

왜 도메인이 필요한가?
컴퓨터는 숫자(IP 주소)로 통신하지만, 사람은 www.google.com 같은 이름이 훨씬 외우기 쉽습니다. 도메인은 사람을 위한 별명이고, 실제 통신은 IP로 이루어집니다.
비유: 전화번호(IP)를 일일이 외우지 않고, 연락처에 "엄마"(도메인)라고 저장하는 것과 같습니다. DNS는 연락처 앱 역할입니다.
개념설명예시
도메인사람이 읽기 쉬운 주소www.google.com
IP 주소컴퓨터가 실제 사용하는 주소142.250.196.110
DNS도메인 → IP 변환 서버전화번호부 역할
HTTP/HTTPS는 뭔가요?
웹 브라우저와 서버가 데이터를 주고받는 규약(약속)입니다. 프로토콜은 프로그램이 아니라 "이런 형식으로 보내고 받자"는 규칙입니다. HTTPS는 여기에 암호화(Security)를 더한 것입니다.
비유: 편지를 보낼 때 "봉투에 보내는 사람·받는 사람을 쓰고, 우표를 붙인다"는 규칙이 프로토콜입니다. HTTPS는 편지를 잠금 금고에 넣어 보내는 것입니다.
프로토콜역할
HTTP웹 통신 기본 규약
HTTPSHTTP + 암호화 (상용 서비스 필수)
TCP/IP인터넷 통신의 기반 규약
① URL 입력 (www.google.com) ↓ ② DNS 변환 (도메인 → IP 주소) ↓ ③ HTTP/HTTPS 요청 (서버에 파일 달라고 요청) ↓ ④ 서버 응답 (HTML + CSS + JS + 이미지 전달) ↓ ⑤ 화면 구성 (브라우저가 조합해서 표시)
💡 최초 요청 파일
브라우저가 가장 먼저 요청하는 파일은 index.html입니다. 이 파일 안에 CSS·JS 파일 경로가 적혀 있어서, 브라우저가 추가로 받아옵니다.
앱은 웹과 뭐가 다른가?
웹은 브라우저가 서버에서 화면(HTML)을 매번 받아오지만, 앱은 화면(프로그램)이 이미 설치되어 있고 서버에서는 데이터(JSON)만 받아옵니다.
비유: 웹은 매번 "책 전체"를 빌려 읽는 도서관이고, 앱은 "책은 이미 갖고 있고" 새 소식만 알림으로 받는 것입니다.

설치 없음 — 브라우저로 접속
서버에서 화면+데이터 모두 받음
업데이트 = 서버만 수정하면 끝
예: 쿠팡 웹사이트, 구글
앱
설치 필요 — 기기에 다운로드
서버에서 데이터만 받음 (JSON/XML)
업데이트 = 앱스토어에서 다시 설치
예: 카카오톡, 엑셀, VS Code
① 앱 실행 (폰/PC에 설치된 프로그램) ↓ ② API 호출 (서버에 데이터 요청 — HTTP/HTTPS) ↓ ③ 서버 처리 (WAS에서 로직 실행 → DB 조회) ↓ ④ JSON/XML 응답 (데이터만 바구니에 담아 전달) ↓ ⑤ 화면 표시 (앱이 데이터를 받아 화면에 표시)

서버가 두 종류인 이유는?
웹서버는 이미 만들어진 파일(HTML, 이미지)을 그대로 전달하는 역할이고, WAS는 요청에 따라 DB를 조회하고 계산해서 결과를 만들어 전달합니다. 하나는 창고지기, 하나는 요리사입니다.
비유: 웹서버 = 편의점 (진열된 상품을 그대로 줌), WAS = 식당 (주문받고 요리해서 줌)
웹서버 — 정적(Static)
대표: Apache, Nginx
하는 일: 고정된 파일 전달
HTML, CSS, JS, 이미지
= 변하지 않는 데이터
WAS — 동적(Dynamic)
대표: Tomcat, Gunicorn
하는 일: 요청마다 계산·조회
DB 연동, 로직 처리
= 매번 바뀌는 데이터
💡 서버 = "대기(Listen)"
서버라는 이름은 "요청을 기다리고 있다(Listen)"는 뜻입니다. 24시간 켜져서 클라이언트 요청이 오면 응답하는 프로그램입니다.

서버 프로그램(백엔드 애플리케이션)은 WAS 위에서 동작하며, 핵심 역할 3가지를 합니다:

#역할설명
1API 개발프론트엔드와 백엔드를 연결하는 통로 만들기
2요청·응답 처리들어온 요청을 분석하고 적절한 결과를 돌려줌
3DB 연동SQL로 데이터를 저장·조회·수정·삭제

왜 데이터베이스에 직접 접근하지 않고 서버를 거칠까?
프론트엔드가 DB에 직접 접속하면 보안이 무너집니다. 누구나 DB를 조작할 수 있게 되니까요. 서버가 중개자(보안 게이트) 역할을 합니다.
비유: 은행 금고(DB)에 고객(프론트)이 직접 들어가면 안 되니, 은행원(서버)이 대신 꺼내주는 것입니다.

데이터베이스 안에는 테이블(Table)이 여러 개 있고, 엑셀 시트와 구조가 같습니다:

DB 용어엑셀 비유설명
Table시트 하나유저 테이블, 상품 테이블 등
Column (열)열 헤더 (A, B, C)이름, 가격, 이메일 등 항목
Record (행)한 줄의 데이터회원 1명, 상품 1개의 정보

← 각 행이 Record, 각 열이 Column

SQL이란?
Structured Query Language — 데이터베이스에서 데이터를 가져오고 저장하는 전용 언어입니다. 프로그래밍 언어가 아니라, DB에게 명령을 내리는 문장입니다.

💡 API ↔ CRUD 연결

① 사용자가 버튼 클릭 (프론트엔드) ↓ ② API 호출 (이미 매핑된 주소로 HTTP 요청) ↓ ③ WAS 수신 (웹 애플리케이션 서버가 요청 받음) ↓ ④ 서버 프로그램 처리 (비즈니스 로직 실행) ↓ ⑤ SQL → DB 조작 (CRUD 실행) ↓ ⑥ JSON/XML 응답 (결과를 바구니에 담아 전달) ↓ ⑦ 화면 업데이트 (프론트엔드가 결과 표시)
🛒 "주문하기" 버튼을 누르면
  1. [프론트] 주문 버튼 클릭 → API 호출 (POST /orders)
  2. [서버] WAS가 요청 수신 → 재고 확인 로직 실행
  3. [서버→DB] SQL: SELECT stock FROM products WHERE id=123
  4. [DB→서버] 재고 5개 있음 → 주문 가능 판단
  5. [서버→DB] SQL: INSERT INTO orders (...) → 주문 기록 생성
  6. [서버→DB] SQL: UPDATE products SET stock=4 → 재고 차감
  7. [서버→프론트] JSON 응답: {"status": "success", "orderId": 456}
  8. [프론트] "주문 완료!" 화면 표시
🎯 자동화 프로그램도 같은 원리
우리 자동화 프로그램은 화면 없는 앱입니다. 사람이 버튼을 누르는 대신, 코드가 자동으로 API를 호출하고, 서버에서 JSON으로 데이터를 받아 처리합니다. 원리는 완전히 동일합니다.

✅ PART 3 핵심 기억 포인트

  1. 웹은 서버에서 화면+데이터를 받고, 앱은 데이터만 받는다
  2. 도메인은 IP의 별명 — DNS가 변환해준다
  3. 웹서버는 창고(정적), WAS는 식당(동적)
  4. DB는 엑셀과 같은 구조 — Table, Record, Column
  5. API 요청(GET/POST) → SQL(SELECT/INSERT) → DB 처리 → JSON 응답

Q. Python 프로그램은 웹이야? 앱이야?
PC에 설치해서 실행하니까 → 앱(데스크톱 앱)입니다.
"앱 = 핸드폰"이 아니라, "앱 = 설치해서 쓰는 모든 프로그램"입니다. 엑셀도, VS Code도, 카카오톡 PC버전도 전부 앱입니다.
핵심 구분법: 브라우저에서 URL로 접속하면 → 웹. 다운받아 설치하면 → 앱. 이것만 기억하면 됩니다.
웹 예시 (브라우저 접속)
www.google.com — 검색
www.coupang.com — 쿠팡 웹
docs.google.com — 구글 문서
스마트스토어 판매자센터
→ 설치 없이 URL만 있으면 됨
앱 예시 (설치형)
핸드폰: 카카오톡, 배민, 쿠팡 앱
PC: 엑셀, VS Code, Python 프로그램
둘 다: 슬랙, 디스코드
→ 기기에 다운로드해서 설치
Q. API가 나오면 앱인가요?
아닙니다. API는 웹이든 앱이든 둘 다 사용합니다. API는 "서버와 대화하는 방법"이지, 웹/앱을 구분하는 기준이 아닙니다.

🎯 우리 자동화 프로그램의 위치

🔑 라이브러리 vs 프레임워크
라이브러리는 내가 호출하고, 프레임워크는 프레임워크가 나를 호출합니다.
→ "Don't call us, we'll call you"

프론트엔드 — 사용자 화면(UI)

  • 사용자가 보고 만지는 모든 것
  • 백엔드에 API 요청·응답 표시

백엔드 — 보이지 않는 로직

  • 비즈니스 로직 처리
  • DB 저장 및 관리
  • 서버 운영
🛒 유통업 예시 — 주문 버튼 클릭
  1. [프론트] 클릭 → 백엔드로 정보 전송
  2. [백엔드] 재고 확인, 결제, DB 저장
  3. [백엔드] 결과를 프론트로 응답
  4. [프론트] "주문 완료" 표시

C언어는 프론트/백 개발용이 아니라, 변수·조건문·반복문·함수라는 프로그래밍의 본질을 배우기 위한 언어입니다.

현대 언어는 라이브러리가 풍부해 본질을 가리지만, C는 메모리·포인터를 직접 다루기 때문에 컴퓨터 동작 원리를 깊이 이해할 수 있습니다.

운영체제(OS)의 역할

  • 프로그램을 실행·관리하는 관리자
  • CPU·메모리·디스크 자원 분배
  • OS도 결국 프로그램의 일종
  • 서버는 대부분 Linux
🔑 서버 프로그램이란
일반 프로그램 앞에 '서버'가 붙으면, 다른 컴퓨터의 요청을 기다렸다가 응답하는 프로그램입니다. (Nginx, MySQL)
  • HTTP: 웹 서버와 클라이언트의 통신 약속
  • 프로토콜은 프로그램이 아닌 '규약'
  • HTTPS: HTTP + Security (암호화)

IP 주소

  • 통신은 근본적으로 IP로 이루어짐
  • 도메인은 IP의 별명 — DNS가 변환
🛒 오픈마켓 API 호출 흐름
내 프로그램 → DNS → 방화벽 → 라우터 → 오픈마켓 서버 → JSON 응답 → 역순 복귀

서버는 한 대가 아니라 역할별로 여러 대로 구성됩니다. 이런 구성을 '아키텍처'라고 부릅니다.

계층역할 / 예시
프레젠테이션UI / React, HTML
애플리케이션로직 / FastAPI
데이터DB / MySQL
🏗️ 왜 계층을 나눌까
  • 유지보수: 화면만 바꿀 때 DB 안 건드림
  • 확장성: 트래픽 많은 계층만 증설
  • 보안: DB 외부 접근 격리

물리 서버 한 대를 소프트웨어적으로 여러 개로 쪼개고 서비스처럼 빌려주는 방식입니다.

  • 필요할 때 5분 만에 서버 생성
  • 쓴 만큼만 비용 지불 (종량제)
  • 서버실 운영 불필요

✅ 핵심 기억 포인트

  1. 모든 기술은 문제 해결의 도구
  2. 자동화는 반복 업무 찾기에서 시작
  3. 가격·재고는 절대 틀리면 안 된다
  4. 작게 시작해 점진적 확장
